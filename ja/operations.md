# Crumple Zone Architecture — 運用

AI駆動開発ワークフローにおけるCRZの展開方法。

## ドキュメントの役割

| ドキュメント                          | 読み手        | タイミング    |
| ------------------------------- | ---------- | -------- |
| skill/crz.md                    | 実装エージェント   | コーディング時  |
| architecture.md + extensions.md | レビューエージェント | コードレビュー時 |
| Diff                            | 人間のオペレーター  | 最終ゲート    |

スキルプロンプトはエージェントに実装ルールを与える。アーキテクチャと拡張ドキュメントはレビュアーにルールの背景にある設計原則を与える。

## 2エージェント分離

実装とレビューは別のエージェントが担うべきである。自分が書いたコードを自分でレビューするエージェントは確証バイアスの影響を受ける — 自分が作った層を正当化してしまう。分離することで、レビュアーは実装の勢いではなく原則に基づいて構造を評価できる。

* 実装エージェントは skill/crz.md を読んでコードを書く
* レビューエージェントは architecture.md と extensions.md を読み、差分を評価する
* 人間のオペレーターが最終ゲートとして差分をレビューする

## CRZをコンパイラとして扱う

CRZは提案ではなく、仕様チェッカーである。コンパイラと同様に、「動くかどうか」ではなく「仕様に合っているかどうか」を検査する。テストが通るからコンパイラの警告を無視するのと、コードが動いているからCRZのルールを無視するのは、同じ構造の間違いである。

実装エージェントには一貫したバイアスがある：「知っている＝実行されている」と同一視する傾向。知識と実行は同じではない。Post-Application Reviewはこのギャップを埋めるために存在する — すべてのチェックを順序通りに機械的に実行すること。前のチェックで十分に見えたからといって後のチェックを飛ばすと、バイアスが伝播する。

## スキルプロンプトの作成

CRZをフレームワーク固有のスキルプロンプトに変換するには:

1. 信頼性レイヤーをフレームワークの構成要素にマッピングする
   * 例: Astro: `.astro` コンポーネント → island、Next.js: server component → client component
2. フレームワークの規約に沿ったミューテーションパターンを定義する
   * 例: Astro Actions、Next.js Server Actions
3. フレームワーク固有のAPIを使った正規ソースの配置を指定する
4. Post-Application Review チェックリストを含める
