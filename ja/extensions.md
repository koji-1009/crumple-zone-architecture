# Crumple Zone Architecture — 拡張議論

設計原則の適用範囲と、それを超える場合のパターンを整理する。

## 1. 設計原則の適用範囲

### MPA + Islands で実現できるもの

サーバーがリクエストに応じてHTMLを返し、対話的な箇所にのみ island component を配置する構成で実現可能なアプリケーション。

* コンテンツサイト、ブログ、ドキュメンテーション
* 管理画面、ダッシュボード
* ECサイト（商品一覧、カート、決済）
* 業務アプリケーション（CRUD、承認フロー）
* SNS のフィード表示（タイムライン、いいね、コメント）

### SPA でしか実現できないもの

クライアントとサーバーが常時双方向に同期し、ブラウザのHTMLレンダリングパイプラインでは対応できないアプリケーション。

* リアルタイム協調編集（Google Docs: Operational Transform / CRDT）
* ベクター/ラスター描画ツール（Figma: WebGL/Canvas エンジン）
* ブラウザ内 IDE（VS Code Web: ファイルシステム仮想化、ターミナル）

これらはMPAの拡張ではなく、別の設計が必要な領域。

### 境界上にあるもの（拡張パターンで対応可能）

MPA を基盤としつつ、部分的にリアルタイム性を持たせることで実現可能なアプリケーション。

* メッセンジャー（チャットメッセージのリアルタイム受信）
* 通知システム（未読バッジ、プッシュ通知）
* ライブダッシュボード（メトリクスの自動更新）
* コメントのリアルタイム反映

## 2. 拡張パターン

### 2.1 島の中のストリーム（小規模リアルタイム）

リアルタイム更新が必要な箇所を一つの island component に封じ込め、Server-Sent Events（SSE）で受信する。

```
Astro API route ──SSE──→ ブラウザ EventSource（ネイティブAPI）
                              ↓
                         island component が受信し表示を更新
```

* SSE の接続管理は EventSource（ブラウザネイティブ）が担う
* 受信データの表示更新は island component 内のローカル状態
* ページの他の部分はサーバーレンダリングのまま
* 接続が切れた場合、EventSource が自動再接続する（ブラウザの仕様）

用途: 通知バッジ、チャットウィジェット、ステータス表示

### 2.2 同期層を正規ソースとして扱う（大規模リアルタイム）

Firebase Firestore のような「ネットワーク + ローカルキャッシュ + 自動同期」を内包するライブラリを導入し、正規ソースとして扱う。

```
Firestore サーバー ←自動同期→ Firestore SDK ローカルキャッシュ
                                    ↓
                              アプリケーションコードは
                              キャッシュから読む（冪等）
```

正規ソースとしての適格性:

* ローカルキャッシュからの読み取りは冪等（何度読んでも同じ値）
* 同期はライブラリが管理し、アプリケーションコードは関与しない
* オフライン時もキャッシュから読める

設計原則との関係:

| 正規ソース          | 管理者       | 同期メカニズム          |
| -------------- | --------- | ---------------- |
| URL            | ブラウザ      | ユーザー操作           |
| Cookie         | ブラウザ      | サーバーの Set-Cookie |
| sessionStorage | ブラウザ      | アプリケーションコード      |
| localStorage   | ブラウザ      | `storage` イベント   |
| 同期層キャッシュ       | ライブラリ SDK | SDK が自動同期        |

いずれも「読み取りは冪等、同期は別のメカニズム」という構造は同じ。

### 2.3 sessionStorage のリアクティブ監視

sessionStorage はページをまたいで UI 状態を保持する正規ソースだが、同一タブ内での変更イベントはブラウザが発火しない。
island から変更を検知するには、書き込みと同時に `CustomEvent` を dispatch し、受信側で listen する。

```typescript
// 書き込み側（island または通常のスクリプト）
sessionStorage.setItem('likeState', JSON.stringify(state));
window.dispatchEvent(new CustomEvent('session-update', { detail: { key: 'likeState' } }));

// 受信側 island
useEffect(() => {
  const handler = (e: CustomEvent) => {
    if (e.detail.key === 'likeState') {
      setState(JSON.parse(sessionStorage.getItem('likeState') ?? 'null'));
    }
  };
  window.addEventListener('session-update', handler as EventListener);
  return () => window.removeEventListener('session-update', handler as EventListener);
}, []);
```

* sessionStorage が正規ソースのままである。CustomEvent は通知手段にすぎない
* island が壊れても sessionStorage の値は残る。次のページロードで正規ソースから再構築できる
* 認証情報を含まない UI 状態（いいねの反映、タブの開閉など）に限定する

用途: ページをまたぐ UI 状態の即時反映

### 2.4 インプレース編集

管理画面などのデータグリッドで、セルをクリックして直接編集するパターン。「どのセルが編集中か」という状態だけを island が持ち、確定時に Action を呼ぶ。セルの表示は Astro コンポーネントのまま、編集中のみ island が介入する。

```
Astro コンポーネント（テーブル全体を表示）
  ↓ クリックイベントを island に委譲
Island（編集中セルの state を管理）
  ↓ 確定時
Astro Action（サーバーで更新、正しさを保証）
  ↓ 成功
リロードでテーブル全体を再構築
```

* 「表示」と「編集状態の管理」を分離することで、island の責務が最小になる
* island が壊れてもテーブルの表示は維持される

### 2.5 地図コンポーネント

Leaflet 等の外部ライブラリはクライアント側での初期化が必須であり、island の正当な使用ケース。座標やマーカーデータはフロントマターで取得してprops として渡す。

```astro
---
const locations = await getLocations(); // サーバーで取得
---
<MapView client:visible locations={locations} />
```

* データ取得はサーバー側で完結し、island はレンダリングのみを担う
* `client:visible` を使い、ビューポート外では JS を読み込まない
* 地図の表示が失敗しても、ページの他の情報は表示されたまま

### 2.6 楽観的な更新

サーバーが処理する間、UIを即座に更新する。リロード後はサーバーデータが上書きするため、状態の権威はサーバーが持ったままになる。

失敗の重大度に応じて2つのエラーハンドリング戦略がある:

**Fire-and-forget** — 失敗しても問題が生じない操作向け（いいね、絵文字リアクション）。2.3 のリアクティブ sessionStorage（sessionStorage + CustomEvent）を利用し、Action をバックグラウンドで送信する。ズレはリロードで解消される。

```
クリック
  → sessionStorage に書き込み（即時反映）
  → CustomEvent で island に通知
  → Action を送信（バックグラウンド）
リロード
  → サーバーデータが正規ソースとして上書き
```

**Revert on error** — ユーザーが永続化を期待する操作向け（ウィッシュリスト、カート、ブックマーク、フォロー）。ローカル状態を楽観的に更新し、サーバーがエラーを返したら元に戻す。

```
クリック
  → ローカル状態を更新（即時フィードバック）
  → Action を送信
  → エラー時: 操作前の状態に戻す
リロード
  → サーバーデータが正規ソースとして上書き
```

両パターンとも正規ソースの権威モデルは同じ。違いは楽観的表示とサーバー確認の間のギャップをクライアントがどう扱うかだけである。

* Fire-and-forget
  * 失敗しても問題が生じない操作に限定する
* Revert on error
  * サーバーが受理か拒否を明確に返せる操作に限定する
  * 不可逆な副作用を伴う操作（課金、外部API呼び出し）には使わない

### 2.7 キャッシュ層

キャッシュ層（CDN、Service Worker、ブラウザキャッシュ）は読み取り側の最適化である。正規ソース（サーバー）と表示の間に位置し、取得済みのデータを提供することでレイテンシの低減やネットワーク中断時の表示継続を実現する。

```
サーバー（正規ソース）
  → キャッシュ層（読み取り側の最適化）
    → ブラウザがページを表示
```

* キャッシュは「最後に正しかった状態」を保持する — クライアントが独自に変更した状態ではない
* キャッシュが古いか存在しない場合、リクエストはサーバーに到達する
* リロードは正規ソースから再構築し、キャッシュの値を上書きする
* キャッシュ戦略（stale-while-revalidate、network-first 等）はパフォーマンスの判断であり、アーキテクチャの判断ではない

キャッシュ層は本質的にクランプルゾーンである。壊れたり古くなっても、サーバーへの次の到達可能なリクエストで修正される。状態の権威がクライアントに移ることはない。

SSR 環境では、サーバーが読み取れる正規ソース（Cookie、データベース）に対するクライアントキャッシュは冗長になることが多い。ナビゲーションのたびに SSR が正規ソースから状態を再構築するためである。クライアントキャッシュが正当化されるのは: (a) 正規ソースがクライアント専用の場合（localStorage）、(b) ネットワークレイテンシを隠す必要がある場合（オフラインファースト）、(c) クライアントサイドナビゲーションが SSR をバイパスする場合（ViewTransition + ClientRouter など）。

例: Firestore は IndexedDB をローカルキャッシュとして使用し、SDK が同期を管理する。キャッシュ層（2.7）としてはオフライン時の読み取りとレイテンシ低減を提供する。同期層（2.2）としては双方向同期によりキャッシュを正規ソースに昇格させる — 接続状況にかかわらず読み取りが冪等になる。同じインフラだがアーキテクチャ上の役割は異なる。

### 2.8 セキュリティ上の注意

sessionStorage と同期層キャッシュは、HttpOnly Cookie と異なり XSS で読み書き可能。設計原則のセキュリティ境界と合わせると:

| 正規ソース            | XSS 耐性          | 置くべきもの            |
| ---------------- | --------------- | ----------------- |
| HttpOnly Cookie  | JS から読めない       | 認証情報、セッション        |
| URL query params | 読める（機密でないことが前提） | フィルタ、ページング        |
| sessionStorage   | 読み書き可能          | 操作途中のデータ（機密でないもの） |
| localStorage     | 読み書き可能          | ユーザー設定（機密でないもの）   |
| 同期層キャッシュ         | 読み書き可能          | 表示用データ（機密でないもの）   |

認証情報は常に HttpOnly Cookie に置く。これは拡張パターンでも変わらない。
