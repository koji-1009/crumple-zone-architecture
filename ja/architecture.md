# Crumple Zone Architecture

ブラウザを信頼し、健全なWebアプリケーションを作るためのアーキテクチャ。「壊れ方」から設計する — 各レイヤーが独立に壊れることを前提とし、障害の影響範囲から逆算する。

かつてフレームワークが再実装していた機能の多くが、モダンブラウザの標準として利用可能になった。このブラウザの発展を、積極的に活用する。
フレームワークによる再実装を最小化し、MPAを基盤として対話的な操作が必要な箇所にのみ、クライアントコンポーネントを配置する。

## 背景

フロントエンドがメンテナンス不能になる主因は、クライアント状態の複雑さの蓄積である。
SPA ではアプリケーション全体がクライアント状態を持つリスク領域になり、時間とともに依存と状態が絡み合う。このアーキテクチャでは状態をサーバーと正規ソースに寄せ、クライアントに残る状態を構造的に制限する。

フロントエンドの作り直しを、現実的な選択肢として維持する。
バックエンドAPIが存在するシステムでは、サーバーはBFFとして薄く保てるため、フロントエンドの技術選定やリプレースがバックエンドに影響しない。
この構造は、現代のWeb、Mobile、Desktopアプリを同一のバックエンドAPIで提供するケースで特に有効となる。

## 優先順位

関心が競合する場合、この順序で選択する。

1. 正しさ > 体験
   * データの整合性とセキュリティは、滑らかな操作感と引き換えにしない
2. 復元可能性 > 継続性
   * 正規ソースから状態を再構築できることを、状態が途切れないことより重視する
3. 可視性 > 抽象化
   * レイヤーの境界がコード構造から目視できることを、フレームワークによる隠蔽より重視する
4. 簡素さ > 豊富さ
   * 層の数とコード量が少ないことを、機能や対話性が多いことより重視する

### 正しさと体験の両立

かつて正しさと体験はトレードオフの関係にあったが、ブラウザの機能向上により多くの問題は解消された。

ページ遷移は `ViewTransition` が滑らかな表現を導入した。フォーム処理は `<form>` + `FormData` が autofill やアクセシビリティと統合された形で提供した。ページをまたぐ UI 状態は `sessionStorage` が活用できるようになった。

フレームワークで再現するのではなく、ブラウザに委ねることで、正しさと体験は両立する。

## 1. 信頼性レイヤー

| レイヤー                    | 障害時の影響         | 設計方針         |
| ----------------------- | -------------- | ------------ |
| HTML / ブラウザAPI          | なし。仕様通りに動く     | 基盤として信頼する    |
| CSS                     | レイアウト崩れ。機能は維持  | 視覚的障害にとどめる   |
| Stateless UI (props のみ) | 入力が誤れば表示が欠ける   | 入力をサーバーで保証する |
| Stateful UI (ローカル状態あり)  | 機能停止。操作不能になりうる | この層を最小化する    |

HTML 層の信頼性はセマンティクスの正しさに依存する。`<button>` はキーボード操作、フォーカス管理、スクリーンリーダー対応をブラウザが提供するが、`<div onclick>` にはいずれもない。適切な HTML 要素を使うことが、この層を信頼できる基盤にする前提条件である。

ブラウザネイティブAPI（Geolocation, Web Speech 等）もこの層に属する。ブラウザの選択はユーザーの責任であり、提供者の責務外である。

設計判断の基準: 「この要素が壊れたとき、何が起きるか」を問い、影響範囲が小さいレイヤーに実装を寄せる。

## 2. 宣言的状態

画面の状態をフレームワークのメモリで管理せず、冪等に読み取れる正規ソースから毎回構築する。

正規ソース（何度読んでも同じ値を返す）:

* URL（パスとクエリパラメータ）
  * ブラウザ管理
* Cookie（HttpOnly）
  * ブラウザ管理。サーバーでも読める
* sessionStorage
  * ブラウザ管理。タブ単位で独立。SPA がフレームワークのメモリに持つ UI 状態を保持する
* localStorage
  * ブラウザ管理。全タブ共有で永続。ユーザー設定（テーマ等）の保持に使う
  * `storage` イベントによりタブ間で自動同期される
* サーバーサイドDB
  * バックエンドが存在する場合のユーザー設定・プリファレンスの権威ある保存先。バックエンドDBがない場合は Cookie や localStorage で代替する
* サーバーデータ
  * 上記をキーにサーバーが引くデータ

過渡的状態（正規ソースではない）:

* UI フレームワークのローカル状態。リロードで消失する

ローカル状態に入れる値は最小化する。フォーム入力では、状態の配置先に選択肢がある:

| 要件              | 実装                    | 状態の所在                                 |
| --------------- | --------------------- | ------------------------------------- |
| 値を送信するだけ        | `<form>` + `FormData` | HTML 層。`<input>` が値を自身で保持する。フレームワーク不要 |
| バリデーション・動的表示が必要 | controlled component  | フレームワーク層。ローカル状態で値を管理する                |
| 入力途中の値を遷移後も保持   | sessionStorage に退避    | ブラウザ管理の正規ソース                          |

controlled component は値を HTML 層からフレームワーク層へ移動させる。信頼性の高い層から低い層への移動であり、明確な理由がある場合にのみ行う。

結果として:

* 画面は正規ソースから決定論的に構築される
* 状態変更は正規ソースの更新とリロードで反映する。リロードは障害ではなく再構築手段である

## 3. セキュリティ境界

クライアントで実行されるアプリケーションコードを非信頼ゾーンとして扱う。サーバーはHTMLを返却するBFFとして機能する。

すべてのアプリケーションに共通する基盤:

* 本番環境では HTTPS を使用
* すべての Cookie: `HttpOnly` + `SameSite=Lax` + `Secure`
* 秘密情報と外部API呼び出しはサーバーに留める

```
ブラウザ（非信頼）
  アプリケーションコードは認証情報を持たない
  外部サービスAPIを呼ばない
  サーバーのAPI routeのみ呼べる
──── HTTPS + HttpOnly + SameSite Cookie ────
サーバー（信頼境界 / BFF）
  Cookie → セッションまたは設定
  バックエンドAPI呼び出し
  秘密情報はここだけ
────────────────────────────────
バックエンドAPI / DB
```

この構造により、XSS が発生しても:

* HttpOnly Cookie は JS から読めない
* SameSite Cookie は外部オリジンに送信されない
* API キーはサーバー内にあり漏洩しない
* UI コンポーネントは認証情報をそもそも受け取っていない

認証が必要な場合、ミドルウェアが毎リクエストで認証をチェックし、未認証ユーザーをリダイレクトする。認証のないパブリックアプリケーションでも、上記の基盤は同様に適用される。

## 4. 体験層

画面遷移と操作フィードバックはクランプルゾーンとして設計する。壊れても機能は動作し、体験が劣化するだけに止める。

`ViewTransition` — MPAのページ切り替えに適用し、SPAのような遷移体験を得る:

* `ClientRouter` をレイアウトに追加するだけでデフォルトの crossfade がページ全体に適用される — 追加のディレクティブは不要
* `transition:animate` や `transition:name` はデフォルトの crossfade には不要。指定するとコンポーネントごとに `view-transition-name` CSS が生成され、個別の調整が必要になる。省略すればこのオーバーヘッドを回避できる
* 非対応ブラウザでは通常のMPA遷移にフォールバックする

操作フィードバック — 壊れても操作自体は完了する:

| 場面   | フィードバック       | 壊れた場合                       |
| ---- | ------------- | --------------------------- |
| 送信中  | ボタン無効化、処理中表示  | 二重送信のリスクがあるが、データは送信される      |
| 成功後  | リロード後にメッセージ表示 | メッセージが出ないが、データは保存されている      |
| 失敗時  | エラーメッセージ表示    | ユーザーが失敗に気づかないが、データは変更されていない |
| 入力検証 | 送信前のバリデーション表示 | サーバー側バリデーションが最終防衛線として機能する   |

フィードバックは二重構造を持つ — クライアントが体験を、サーバーが正しさを担う。クライアント側が壊れても、サーバー側が機能を保証する。

## 5. 判断フロー

### 5.1 実装レイヤーの選択

```
ユーザー操作で表示が変わるか？
├─ No → サーバーレンダリング（HTML）
└─ Yes → ページ遷移で解決できるか？
          ├─ Yes → リンク（<a>）で遷移
          └─ No → ローカル状態が必要か？
                   ├─ No → <script>（DOM操作のみ: dialog.showModal()、スクロール、クリップボード）
                   └─ Yes → クライアントコンポーネント
                             ローカル状態を最小化し、
                             状態を持たない子に分離する
```

### 5.2 状態の配置先

認証の有無ではなく、状態の種類で判断する:

```
どの種類の状態か？
├─ 表示状態（フィルター、ページネーション、ソート、検索、アクティブタブ）
│                                 → URL query params（認証の有無に関わらず常に）
├─ 認証状態（セッション）         → Cookie (HttpOnly)
├─ 設定・好み（テーマ、デフォルト値）
│   ├─ バックエンドDBがある       → サーバーサイドDB
│   ├─ サーバーで読む必要がある   → Cookie (HttpOnly)
│   └─ クライアントのみ           → localStorage
├─ タブ単位の作業中データ         → sessionStorage
└─ いずれにも該当しない（送信まで一時的）
                                  → コンポーネントのローカル状態（過渡的）
```

URLパラメータとcookieのデフォルト値が重複する場合（例: 検索対象の好みがcookieにあるが、URLに`?target=x`がある）、URLパラメータが優先される。

### 5.3 サーバーレンダリング: 同期と遅延

サーバーレンダリングされるコンポーネントは、同期的（ページ読み込みをブロック）または遅延的（フォールバックコンテンツとともに非同期読み込み）に配信できる。

同期（デフォルト）:

* ページの主コンテンツ — ユーザーに届く前に配信を保証する
* 不在がユーザーを誤解させるコンテンツ
* 複数ソースから取得する場合は部分失敗パターンを使用する

遅延:

* ユーザーがローディングを自然に受け入れる、計算コストの高い処理
* キャッシュ可能なページ上のパーソナライズされたコンテンツ

遅延レンダリングはクランプルゾーンである。失敗した場合、フォールバックが残り、ページは機能し続ける。フォールバックは単なるローディング表示ではなく、それ自体で意味のあるコンテンツとして設計する。

### 5.4 ブラウザAPI成熟度の評価

「ブラウザを信頼する」は無条件ではない。ブラウザAPIに委ねる前に、4軸で評価する:

| 軸 | 信頼可能 | 未成熟 |
| --- | --- | --- |
| クロスプラットフォーム互換性 | デスクトップ、モバイル、支援技術で一貫した動作。軽微な視覚差は許容、動作差は不可 | ブラウザ間で動作が異なる、または主要プラットフォームで未実装 |
| 合成可能性 | 標準的なCSS・HTML・JSパターンと自然に組み合わせられる | 機能するために非自明な回避策が必要。CSSでスタイルできない。標準のイベントモデルを無視する |
| 障害モードの透明性 | 動作する（機能は落ちる）状態へのグレースフルデグラデーション。機能検出が容易 | サイレント障害 — 動いているように見えるが、不正確または不整合な結果を出す |
| 仕様の安定性 | WHATWG Living Standard または W3C Recommendation。Baseline Widely Available | フラグの背後、設計変更中、または出荷後に仕様から削除 |

**4軸すべてで信頼可能**と評価されたAPIのみ、直接委任できる。いずれか1軸でも不合格なら封じ込めが必要。

封じ込め戦略（優先順）:

1. **回避** — そのAPIを必要としない代替パターンを使う
2. **隔離** — 単一コンポーネントに閉じ込め、契約を定義する。コンポーネントが自身のフォールバックを処理する。アプリケーションの他の部分はAPIに直接触れない
3. **ライブラリへの委任** — 専用ライブラリを使うが、プロジェクト所有のインターフェースで包む。機能コードがライブラリを直接importしない

3つの故障パターンが対応戦略と再評価の時期を決定する:

| パターン | 原因 | 例 | 戦略 | 再評価 |
| --- | --- | --- | --- | --- |
| 構造的設計欠陥 | 仕様自体が誤っている | Drag and Drop API（マウス専用設計） | 恒久的に回避 | 新仕様が置き換える場合のみ |
| 実装の遅れ | 仕様は妥当だがエンジンが未出荷 | `appearance: base-select` | 一時的に隔離 | 2つ目・3つ目のエンジンが出荷した時 |
| 仕様の不足 | 仕様が動作を意図的に未定義にしている | `<datalist>` のフィルタリングアルゴリズム | 非自明な用途では回避 | 仕様が動作を定義した時 |

3つ目のパターンが最も厄介である。機能検出を通過し、動作しているように見え、ブラウザごとに異なる動作をする — すべて仕様の範囲内で。

個別APIの評価とClientRouterの移行計画は `references/` を参照。

## 6. 前提

1. ブラウザを信頼する — 成熟度に応じて（5.4）。未成熟なAPIはクランプルゾーンで封じ込め、欠陥は上流で修正してゾーンを薄くする。MPA の適用範囲はブラウザの機能拡充とともに拡大し続けている
2. 壊れ方を設計する。機能追加ではなく、障害封じ込めから設計を始める
3. サーバーが状態の権威である。クライアントは表示層であり、リロードは正規ソースからの再構築手段である
4. セキュリティ境界を目視できるようにする。暗黙のRPCではなく、明示的な API route とする
